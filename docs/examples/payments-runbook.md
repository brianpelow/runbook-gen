# Payments Service — Operational Runbook

> Example runbook generated by runbook-gen

## Service overview

The payments service handles ISO 20022 payment message processing for
retail and institutional clients. It is a SOX in-scope system.

## On-call contact

- Primary: payments-oncall@example.com
- Escalation: platform-eng@example.com

## Common alerts and responses

### HIGH_PAYMENT_LATENCY

**Trigger**: P99 latency > 2000ms for 5 minutes
**Impact**: Delayed payment processing, potential SLA breach
**Steps**:
1. Check database connection pool utilization
2. Review recent deployments in the last 2 hours
3. Check downstream FX rate service health
4. If latency > 5000ms, trigger circuit breaker

### PAYMENT_FAILURE_RATE_HIGH

**Trigger**: Failure rate > 2% over 10 minutes
**Impact**: Revenue impact, regulatory reporting required
**Steps**:
1. Identify failure category from error logs
2. Check upstream message broker connectivity
3. Verify ISO 20022 schema validation is passing
4. Escalate to compliance if failures exceed 1000

## Rollback procedure

1. Identify last stable deployment tag
2. Run: `kubectl rollout undo deployment/payments-service`
3. Verify health check passes
4. Notify change management within 30 minutes (SOX requirement)

## Compliance notes

- All incidents must be logged in the change management system within 1 hour
- Payment failures exceeding thresholds require regulatory notification
- Runbook must be reviewed quarterly per FFIEC guidance
